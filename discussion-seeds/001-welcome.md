---
title: "Welcome: what should Codex Dev Loop become?"
category: "General"
---

Codex Dev Loop starts as a small, file-based operating system for long-running Codex desktop development work.

The first release focuses on:

- project memory in `.codex-loop/`
- strong goals with verification
- approval queues for irreversible actions
- automation/runbook patterns
- dashboard generation

Useful discussion prompts:

- What should be easier to install?
- What should the dashboard show first?
- Which workflow should get a dedicated runbook next?
- What would make this safer for real production repos?
